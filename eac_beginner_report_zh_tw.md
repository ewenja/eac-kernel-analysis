# EAC 總報告（新手也看得懂版）

> 這份是給第一次碰 Easy Anti-Cheat 的人看的總整理。  
> 不走論文口氣，盡量講人話，但內容還是以這個 repo 的靜態分析，加上近期官方資料與社群研究交叉整理而成。

---

## 一句話先講完

EAC 厲害的地方，不是只有某一招很猛，而是它把很多檢查疊成一整片安全網。

它會看：

- 你的遊戲有沒有被改
- 有沒有奇怪的 process 在碰遊戲
- 有沒有可疑 driver 載進 kernel
- 記憶體長得正不正常
- hardware ID 是不是怪怪的
- user-mode 跟 kernel-mode 之間講的話可不可信
- 回傳到後端的遙測資料夠不夠完整

所以對新手來說，最容易搞錯的一點就是：  
**EAC 不是只靠掃記憶體在做事。**

---

## 1. EAC 到底是什麼

Easy Anti-Cheat 是 Epic 旗下的一套反作弊系統，現在算是 Epic Online Services（EOS）生態的一部分。  
它不只是「抓到作弊就 ban」，比較像是：

1. 先把作弊變得很難做
2. 再把可疑訊號持續收集起來
3. 最後交給本地規則或後端一起判斷

官方講法也很保守，他們不會說自己一定擋得住所有作弊。這反而是正常的，因為反作弊本來就是攻防，不是裝上去就天下無敵。

---

## 2. 它大概怎麼運作

你可以把 EAC 想成兩層：

### 第一層：使用者模式（Ring 3）

也就是 `EasyAntiCheat.exe` 或 `EasyAntiCheat_EOS.exe` 這類程式。

它主要做的事有：

- 啟動或協助啟動 driver
- 跟遊戲、EAC SDK、driver 當中介
- 收集使用者模式能看見的資訊
- 把一些資料送去 EAC 後端
- 定期送 heartbeat，證明自己還活著

### 第二層：核心模式（Ring 0）

也就是 `EasyAntiCheat.sys`。

這層才是整個系統最硬的地方，因為它跟 Windows kernel 同權限，很多 user-mode 的攔截、hook、偽裝，在這層都沒那麼好用。

它會去碰的範圍大概包括：

- `EPROCESS`
- VAD tree
- handle table
- driver / module 列表
- 硬體資訊
- 憑證與簽章
- 遙測封包組裝

---

## 3. 為什麼大家一直說它很難逆

簡單講，因為它不只藏資料，連自己怎麼叫 API 都藏。

這份 repo 裡很明顯可以看到幾件事：

- 沒有正常 import table
- 函式位址是執行時解出來的
- 字串是混淆過的
- 很多邏輯用 SIMD、虛擬化或拆碎控制流包起來

對分析的人來說，最煩的不是單一函式多長，而是你常常連「這段到底在叫哪個 Windows API」都要先解謎。

而且近年的社群文章也提到，**runtime dump 跟硬碟上的靜態檔案可能差很多**。  
也就是說，你看到的 `.sys` 檔，未必就是它跑起來的完整樣子。

---

## 4. EAC 最核心的幾種偵測思路

### 4-1. 看誰在碰遊戲

這是最直覺的一層。

如果有別的 process：

- 對遊戲開可疑 handle
- 讀寫遊戲記憶體
- 有奇怪的父子 process 關係
- 做出很像 overlay 的視窗

那 EAC 就會開始盯。

### 4-2. 看遊戲記憶體裡有沒有怪東西

這邊重點是：

- VAD tree 掃描
- 可執行記憶體區域
- 手動映射 DLL
- RWX 頁面
- shellcode 類型的痕跡

也就是說，就算你不走傳統 `LoadLibrary`，EAC 也不是完全沒東西看。

### 4-3. 看 kernel 裡有沒有奇怪 driver

這層是很多人覺得它麻煩的地方。

因為一旦可疑程式跑到 kernel，EAC 也會盡量去看：

- 模組有沒有正常列在清單上
- code section 有沒有被 patch
- driver 有沒有簽章問題
- dispatch table / SSDT / inline hook 有沒有異常

簡單講，它不只是在抓 user-mode 外掛，也在防有人直接下去 kernel 搞事。

### 4-4. 看硬體身分是不是有問題

如果帳號被 ban，有些人會去改硬體識別，想假裝自己是另一台電腦。

EAC 這邊不是只讀一個值，而是會交叉看很多來源，像是：

- 磁碟序號
- volume 資訊
- MAC 位址
- GPU 裝置資訊
- SMBIOS / 韌體資訊

所以 spoofing 難搞的點，不是改一個值，而是你得讓很多來源一起看起來合理。

### 4-5. 看回報資料本身是不是可信

這邊很多新手比較少注意到。

EAC 不只是「掃完就算了」，它還會組裝遙測資料、壓縮、簽名，再交給 user-mode 或後端。

重點不是它有沒有用某個特定演算法，而是：
**它很在意資料是不是被中途改過。**

---

## 5. 它最強的地方在哪

如果要我用最白話的方式講：

### 最強的不是單點，是交叉驗證

例如同一件事，它可能不只看一條線：

- 你說這個 process 正常  
  它可能去看 image 名稱、父 process、簽章、handle 行為

- 你說這個 driver 沒問題  
  它可能去看模組列舉、section 權限、hash、憑證

- 你說這台機器是新的  
  它可能去對多個硬體來源

這種設計的麻煩點就在於：
你繞掉一層，不代表你真的過關。

## 6. 近年的外部觀察，哪些值得信

這次我另外補查了近一兩年的資料後，覺得下面幾點值得收：

### 6-1. 不同遊戲、不同 build，可能真的差很多

這點很重要。

不要以為：

- A 遊戲看到的常數
- B 遊戲看到的 IOCTL
- C 遊戲看到的混淆參數

一定都一樣。

比較合理的理解是：
EAC 有一套共同架構，但實際落地時可能會依遊戲、版本、風險等級做調整。

### 6-2. 靜態檔案不一定代表全部

近年的 runtime 研究有提到：

- 載入後體積可能變大
- 虛擬化區段可能展開
- 匯入解析參數可能跟靜態樣本不同

這代表如果只看檔案本身，很可能會低估它實際跑起來的複雜度。

### 6-3. 它可能不只看記憶體，也開始往輸出層看

這點現在還不能講死，但很值得注意。

有些近年的分析看到 EAC 解析到跟 Win32k / GDI 有關的函式，像是：

- `GetDC`
- `CreateCompatibleDC`
- `CreateCompatibleBitmap`
- `BitBlt`
- `GetDIBits`

這些函式如果單看用途，確實會讓人懷疑它跟：

- 畫面擷取
- 畫面比對
- overlay 檢查

有關。

但目前比較穩的寫法應該是：

> EAC 有往輸出層看的線索，但還不能直接斷言每款遊戲都在實際截圖，也不能斷言一定會上傳。

這樣比較準。

## 7. 哪些地方不要太快下結論

### 7-1. 不要看到單篇論壇文就當成全部真相

論壇文很有用，但也很常出現這幾種問題：

- 只針對單一 build
- 只看靜態沒做動態
- 作者自己也說還沒驗完
- 留言區混了很多嘴砲和腦補

所以比較好的讀法是：

- 把它當線索
- 跟 repo 裡的靜態分析互相比
- 再拿官方或 Windows 文件去校正

### 7-2. 不要把「看到 GDI 路徑」直接寫成「EAC 一定天天截圖」

這樣太滿。

合理說法是：

- 看起來有這個方向
- 用到的 API 很像畫面擷取鏈
- 但啟用條件、涵蓋遊戲、是否上傳、是否只是備用路徑，都還不能下死結論

### 7-3. 不要把固定 key、固定常數當通則

如果某篇分析抽出某個 AES key、某個 resolver modulus、某組 IOCTL code，那比較像：

- 某個時間點
- 某個樣本
- 某個 build

看到就知道「這個樣本曾經長這樣」就好，不適合直接寫成 EAC 的永遠定義。

---

## 8. EAC 的弱點是不是很多

要老實講，**有弱點，但不代表好打。**

這份 repo 的弱點章節列了不少缺口，可是很多都屬於：

- 需要高權限
- 需要很熟 Windows kernel
- 需要多層技巧配合
- 稍微做錯就直接藍屏

所以不能用「有缺口」直接推論成「很好繞」。

比較準的講法是：

> EAC 不是完美，但它把攻擊門檻墊得很高，尤其對只會做單點技巧的人很不友善。

## 10. 最後一句總結

如果你問我，這份分析讀完後，對 EAC 最準的一句話是什麼？

我會說：

> EAC 真正難纏的地方，不是它知道所有事，而是它會把很多你以為不重要的小訊號，慢慢拼成一張足夠有用的作弊地圖。

也因為這樣，研究 EAC 最重要的不是只盯著某個 hook、某個頁面、某個 IOCTL，而是要一直問：

**它這一層資料，最後會不會跟別層資料連起來？**

只要這個問題你開始有感覺了，你就真的進入這份 repo 想帶你看到的世界了。
